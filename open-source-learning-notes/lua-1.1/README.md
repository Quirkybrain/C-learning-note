# Lua 1.1 学习笔记

这份目录记录了我对早期 `Lua 1.1` 源码的一次集中学习、编译适配与局部修改实践。

它不仅保留了原始项目源码，也包含了我在阅读解释器实现过程中的分析文档、问题复现脚本，以及针对历史版本缺陷做的实验性修复验证。

## 这份笔记主要做了什么

- 阅读并整理 `Lua 1.1` 的整体执行流程，包括词法分析、语法分析、字节码生成与解释执行
- 分析早期 Lua 的核心数据结构，例如 `Object`、虚拟栈、符号表、字符串表与 `Table/Hash`
- 在现代 Linux / GCC 环境下重新整理构建方式，使历史代码能够更顺畅地编译和调试
- 复现 `Table` 读操作错误创建 `nil` 节点导致的内存膨胀问题，并对相关行为进行修改与验证
- 尝试为 Lua 1.1 增加额外的注释兼容测试，记录个人改动过程

## 目录说明

- [`技术剖析.md`](./技术剖析.md)：本目录最核心的学习文档，记录环境搭建、源码结构、关键机制分析与 bug 修改思路

## 这份笔记关注的几个重点

### 1. 早期 Lua 的解释器实现方式

`Lua 1.1` 的代码量不算大，但已经能看到很多后来 Lua 设计的雏形，比如：

- 动态类型统一表示
- 基于虚拟栈的执行模型
- 字符串驻留
- `Table` 作为核心数据结构
- 简洁但完整的垃圾回收流程

这也是这份学习笔记最值得反复看的部分。

### 2. 历史代码在现代环境下的兼容问题

由于项目年代较早，直接在现代环境中编译会遇到一些典型问题，例如：

- 旧式 `Makefile` 与工具链参数不兼容
- 部分头文件和初始化方式已经不适用于当前编译器
- 默认构建选项不利于调试和继续修改

这些处理细节已经在 [`技术剖析.md`](./技术剖析.md) 中做了较完整的记录。

### 3. 一个很有代表性的旧实现缺陷

本目录中比较有意思的一部分，是围绕 `Table` 读取逻辑展开的分析：

- 旧实现中，读取不存在的键时可能顺手创建一个值为 `nil` 的节点
- 这种行为在“只读访问”场景下会悄悄改变表结构
- 持续读取不存在的索引时，会制造大量无意义节点并持续消耗内存

[`attack.lua`](./attack.lua) 就是为了演示这个问题而准备的。

## 建议阅读顺序

1. 先看 [`技术剖析.md`](./技术剖析.md)，建立对整体结构和修改背景的理解
2. 再看 [`attack.lua`](./attack.lua) 与 [`comment_check.lua`](./comment_check.lua)，理解我具体做了哪些实验和验证
3. 最后进入 [`lua/src/`](./lua/src/) 对照源码阅读，会更容易抓住重点
